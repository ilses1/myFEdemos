# 收益图叠加点逻辑调整计划

## 目标

修改 [incomeChart/index.tsx](file:///c:/Users/wusan/Desktop/myFEdemos/src/pages/Chart/incomeChart/index.tsx) 的「时间区间选择后归 0 与叠加点」逻辑，使其满足：

- 选择某个时间区间之后：
  - 归 0 基准需要相对所选开始日期往前推 1 个交易日
  - 叠加点也需要相对所选开始日期往前推 1 个交易日
- 叠加点取点的前置条件：
  - 取叠加点时要取“所有券都有数据”的第一个点
  - 若“前一个交易日”数据不全，则依次往后推，直到找到全量数据的第一个日期

说明：当前页面的“叠加券”收益序列为 mock 生成（并非真实逐券数据），但会按“可缺失数据”的方式重构计算链路，以支持后续真实数据接入时不改逻辑。

## 现状梳理（关键点）

- 当前时间区间过滤：直接用 RangePicker 的 `[start,end]` 过滤主线数据（line1/line2），并把过滤后的第 0 个点作为归 0 基准（`buildSeries(0)`）。
- 归 0 基准切换：仅由 dataZoom 的 startValue/start 推导 `baseIndex`，再用 `percentFromBaseIndex(..., baseIndex)` 与 `rebaseBySubtract(..., baseIndex)` 重算。
- 叠加点：`overlay-point` scatter 永远画在 `dates[baseIndex]` 且 y=0。

问题：当用户选择时间区间后，基准点被固定为区间起点（索引 0），并不会“往前推一个交易日”；同时没有“所有券都有数据”的校验与回退推进逻辑。

## 设计原则（本次改动的可预期行为）

1. RangePicker 的展示值仍保持用户选择的 `[start,end]`（不改 UI 显示），但图表内部会基于该选择计算一个“基准日期 baseDate”，用于：
   - 初始归 0 基准（baseIndex=0 对应 baseDate）
   - 初始叠加点位置（叠加点落在 baseDate 上）
2. baseDate 的计算流程：
   - 先取候选日期 `candidate = 所选 start 往前的第一个交易日`
   - 从 candidate 开始向后扫描，找到“所有当前展示的券/系列”都存在数据的最早日期，作为 baseDate
   - 若扫描至 end 仍找不到，则回退为“主线可用的最早日期”（保证页面不崩溃，并保持可视化输出）
3. 图表显示的数据区间将从 baseDate 起（而不是用户选择的 start 起），确保叠加点/归 0 基准都能落在可见的 xAxis 上。

## 实现方案

### 1) 抽出交易日/可用性工具函数（在 index.tsx 内部即可）

- `buildDateValueMap(items) -> Map<date, value>`
- `getPrevTradingDate(allSortedDates, startDateStr) -> string | null`
  - allSortedDates 可直接使用主线数据中的日期序列（已有排序）
  - 交易日定义：数据里存在的日期即视为交易日（无需接入交易日历）
- `findFirstCompleteDate(dates, requiredSeriesMaps) -> string | null`
  - 从 dates[0] 开始遍历，若对所有 series map 都 `has(date)`，返回该 date

requiredSeriesMaps 的组成：

- 主线两条（line1/line2）必选
- overlaySelected 非空时，将每条叠加线也纳入 required（从其 date->value map 判断）

### 2) 调整“处理数据 + 初始化 baseIndex”逻辑

当前 `useEffect` 内部处理顺序是：

- `getProcessedData()` 直接过滤所选 `[start,end]`
- `dates = line1.map(...)`
- `series: buildSeries(0)` 且 `baseIndexRef.current=0`

改为：

1. 仍读取用户选择的 `pickedStart/pickedEnd`
2. 用全量日期序列计算 `candidateStart = prevTradingDay(pickedStart)`（如果不存在则用 pickedStart）
3. 在 `[candidateStart, pickedEnd]` 范围内生成候选日期序列 candidateDates（来自主线日期序列过滤）
4. 为主线与叠加线构建 date->value map（叠加线先按候选日期生成/对齐）
5. 用 “所有 series 都有数据” 的规则在 candidateDates 中求 `baseDate`
6. 将最终绘图日期轴 `dates` 设为 `candidateDates` 从 baseDate 开始的切片（从而 baseIndex 固定为 0）
7. 以该 dates 重算：
   - 主线：按 baseDate 取基准值，输出 `((v-base)/base*100)`，缺失日期输出 null（保持折线断点更直观）
   - 叠加线：按 baseDate 取基准值，输出 `v-base`，缺失日期输出 null
8. 初始 setOption 用 `buildSeries(0)`，并把 dataZoom start 设为 0（保持现有交互）

### 3) 兼容 dataZoom 拖动的基准切换

保留现有 dataZoom 事件逻辑（`startValue/start -> nextBaseIndex`），但需要保证：

- `dates` 已经是“从 baseDate 起”的轴；因此初始 baseIndex=0 对应 baseDate
- 拖动后 baseIndex 变化时，仍以该 index 对应的日期作为新基准
- 归 0 与叠加点始终使用同一个 baseIndex（保持一致性）

### 4) 行为校验（手工验证点）

在本地 dev server 下验证：

- 选择任意区间（RangePicker 与预设）后：
  - 归 0 起点（0%）对应日期 = 所选 start 的前一个“数据存在的交易日”（或满足全量数据后的第一个日期）
  - 叠加点落点日期与归 0 基准日期一致
- 当叠加券数量 > 0：
  - 若模拟制造某条叠加线在 candidateStart 当天缺失（可临时在 mock 生成处跳过一个日期），基准日期会自动往后推到全量数据的第一个日期
- dataZoom 拖动后：
  - 叠加点随起点移动
  - 主线与叠加线归 0 基准同步切换
